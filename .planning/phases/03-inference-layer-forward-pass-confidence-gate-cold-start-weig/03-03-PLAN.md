| config.inference.weights (developer/build-time config) -> forwardPass | build-time-injected data, not end-user input, but malformed shape could still crash or silently corrupt inference |
| signal:detected payload -> oneHot -> forwardPass input | payload.type is drawn from a fixed, closed set of 4 strings already validated by Phase 2's signal.js — no new external input surface this phase |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02 | Tampering / Denial of Service | `config.inference.weights` shape (malformed/non-numeric entries, wrong dimensions) | medium | accept | `config.inference.weights` is developer/build-time-injected config, never raw end-user input (RESEARCH.md Security Domain V5); a shape-validation extension to `config.js`'s existing CFG-02 hard-fail pattern is cheap defense-in-depth but explicitly out of this phase's v1 harness scope per RESEARCH.md — accepted for now, flagged for future hardening before a real partner pilot |
| T-03-03 | Denial of Service | `softmax`'s `Math.exp()` overflow on unbounded/extreme logits | medium | mitigate | Max-subtraction implemented in `softmax()` (Task 1) — the universally standard one-line fix for a 4-element vector, verified via INF-01/INF-02 tests passing with no NaN/Infinity in output |
| T-03-06 | Information Disclosure | `inference:result`'s additive pass-through fields (bbox/targetSelector/scrollDepth/pathname) | low | accept | These fields already passed Phase 2's No-PII firewall (`signal.js`'s `buildPayload` choke-point, SIG-05) before ever reaching `inference.js` — this plan forwards already-sanitized geometry/timing data one hop further, reads no new raw DOM/user field |

### `app/swap/page.tsx` (Screen 2 — Client Component, needs Suspense wrapper)

**Source:** RESEARCH.md Pattern 1 (verbatim, adapted from Next.js `useSearchParams` docs)

```tsx
'use client'

import { useState } from 'react'
import { useRouter, useSearchParams } from 'next/navigation'
import { Input } from '@/components/ui/input'
import { Button } from '@/components/ui/button'

export default function SwapAmountForm() {
  const router = useRouter()
  const searchParams = useSearchParams()
  const [amount, setAmount] = useState(searchParams.get('amount') ?? '')

  return (
    <>
      <Input
        data-heed="amount-input"
        type="number"
        inputMode="decimal"
        value={amount}
        onChange={(e) => setAmount(e.target.value)}
      />
      <div data-heed="fee-row">Fee: 0.30%</div>
      <div data-heed="min-received-row">Min received: {/* placeholder, D-01 */}</div>
      <Button
        data-heed="proceed-cta"
        onClick={() => router.push(`/swap/confirm?amount=${encodeURIComponent(amount)}`)}
      >
        Proceed
      </Button>
    </>
  )
}
```

**Critical constraint (Pitfall 1):** This component calls `useSearchParams()` and MUST be wrapped
in `<Suspense fallback={...}>` inside `app/swap/page.tsx`'s exported default page, or `next build`
will fail (works fine in `next dev`, fails only at build time — always run a production build to
verify).

### `app/swap/confirm/page.tsx` (Screen 3 — Server Component, async `searchParams`)

**Source:** RESEARCH.md Pattern 2 (verbatim, adapted from Next.js `searchParams` page-prop docs)

```tsx
export default async function ConfirmPage({
  searchParams,
}: {
  searchParams: Promise<{ amount?: string }>
}) {
  const { amount } = await searchParams
  return (
    <>
      <p>Amount: {amount}</p>
      <div data-heed="fee-row">Fee: 0.30%</div>
      <div data-heed="min-received-row">Min received: {/* placeholder */}</div>
      <a data-heed="back-btn" href={`/swap?amount=${amount}`}>Back</a>
      <a data-heed="confirm-cta" href="/swap/success">Confirm</a>
    </>
  )
}
```

**Critical constraint (Pitfall 2):** `searchParams` is a `Promise` in current Next.js (App
Router, v15+) — must `await searchParams` inside an `async function Page(...)`, never destructure
it as a plain object.

**Note on UI-SPEC.md deviation:** UI-SPEC.md's Screen-by-Screen Content Contract clarifies that
Screen 3's `fee-row`/`min-received-row` values must be rendered as **plain, unattributed rows**
(not carrying `data-heed="fee-row"`/`"min-received-row"` a second time) since those selectors are
Screen-2-owned per CONTRACT.md. The RESEARCH.md example above (which does attribute them on
Screen 3) should be adapted: keep the static values but drop the `data-heed` attributes on Screen
3's copies.

### `components/ui/{button,input,card}.tsx`

**Source:** Do not hand-write. Generate via `npx shadcn add button input card` after `npx shadcn
init`. RESEARCH.md verified (against upstream `shadcn-ui/ui` source,
`apps/v4/registry/new-york-v4/ui/{button,input}.tsx`) that both `Button` and `Input` spread
`{...props}` — including arbitrary `data-*` attributes — directly onto the native `<button>`/
`<input>` element, provided `asChild` is left `false`:

```tsx
// Source: shadcn-ui/ui button.tsx (verified upstream)
function Button({ className, variant, size, asChild = false, ...props }) {
  const Comp = asChild ? Slot.Root : "button"
  return (
    <Comp
      data-slot="button"
      data-variant={variant}
      data-size={size}
      className={cn(buttonVariants({ variant, size, className }))}
      {...props}   // <-- data-heed="proceed-cta" lands here, on the native <button>
    />
  )
}
```

This means every locked selector (`amount-input`, `proceed-cta`, `confirm-cta`, `back-btn`) can
be passed as a plain `data-heed="..."` prop directly on the shadcn component — no wrapper
needed, no raw HTML fallback required.

## Shared Patterns

### Selector placement (all 4 screens)
**Source:** RESEARCH.md Pattern 3 + Pitfall 3
**Apply to:** every element carrying one of the 7 locked `data-heed` selectors
- Apply `data-heed="..."` directly on the shadcn primitive (`<Input data-heed="amount-input" />`,
  `<Button data-heed="proceed-cta">`), never on a custom wrapper component around it.
- Never use `asChild` on an element that also carries a `data-heed` attribute — if `asChild` is
  used (e.g. `<Button asChild><Link>...</Link></Button>` per Screen 4's Done CTA and Screen 1's
  Swap CTA, neither of which carry locked selectors so this is safe there), the `data-heed` prop
  must go on the inner element, not `Button` itself.

### Query-string as the only state channel
**Source:** RESEARCH.md D-02/SCRN-06, Anti-Patterns
**Apply to:** `app/swap/page.tsx`, `app/swap/confirm/page.tsx`
- Only `?amount=X` travels between screens. Never read/write `localStorage`/`sessionStorage`, and
  never keep the amount only in component state across a navigation — it must round-trip through
  the URL (via `useSearchParams`/`router.push` on the client, or the `searchParams` page prop on
  the server).

### No calculation logic (D-01 boundary)
**Source:** CONTEXT.md D-01, RESEARCH.md Anti-Patterns
**Apply to:** `app/swap/page.tsx`, `app/swap/confirm/page.tsx`, `lib/swap-constants.ts`
- `fee-row`, `min-received-row`, and the gas estimate are static strings/values pulled from
  `lib/swap-constants.ts` with zero recompute logic. Do not derive fee/min-received from `amount`
  in this phase — that is explicitly Phase 2's job (CALC-01/02/03).

### Suspense boundary for `useSearchParams`
**Source:** RESEARCH.md Pitfall 1
**Apply to:** `app/swap/page.tsx` only (the sole Client Component using `useSearchParams`)
- Wrap the client form component in `<Suspense fallback={...}>` in the page's default export, or
  `next build` will fail even though `next dev` succeeds.

  <action>Pause for human verification — the operator opens test-harness/index.html in a real mobile-viewport (390px) browser and confirms the response overlay renders, positions, animates, and logs per 04-UI-SPEC.md using the how-to-verify steps below. No files are modified; this is a blocking gate.</action>
  <what-built>
    Response overlay (src/response.js) rendering 4 confidence-gated bubble types (tooltip, nudge_copy, discount_offer, social_proof) into a single fixed full-viewport pointer-events:none container, positioned via safe-area-aware clampToViewport, plus structured [heed] logging (src/log.js) for all 6 pipeline events and session-lifecycle wiring — all wired into init() (src/index.js) and built into dist/sdk.js.
  </what-built>
  <how-to-verify>
    1. Run `npm run build` (if not already fresh from Plan 04-05), then open `test-harness/index.html` in a browser with a 390px mobile viewport (Chrome DevTools device toolbar → iPhone 12/13 or custom 390px width).
    2. Open the browser console. Trigger each debug-panel signal button in turn (touch_hesitation, blur_incomplete, scroll_reversal, back_intent).
    3. For each, confirm a response bubble renders with the correct copy per 04-UI-SPEC.md's Response Types table: confusion→tooltip ("Not sure what this means? Tap for a quick explanation."), flow_friction→nudge_copy ("Almost there — one more step to finish."), price_doubt→discount_offer (body "Complete now and save on fees." + "See offer" CTA), trust_gap→social_proof ("Thousands of people completed this safely today.").
    4. Confirm colors match UI-SPEC (dark #1a1a1a bubble, white text, #2f6fed accent only on the discount_offer CTA and the tooltip caret), the entrance animation (fade + slide-up ~200ms), and the 6000ms auto-dismiss for tooltip/nudge_copy/social_proof (discount_offer persists until dismissed).
    5. Confirm the overlay does NOT block the host page: while a bubble is showing, tap/scroll the host content underneath and confirm it still responds; then tap the bubble's × (44×44 hit target) and the discount_offer CTA and confirm each is itself tappable and dismisses correctly.
    6. Confirm single-bubble concurrency (D-05): trigger a second signal while a bubble shows — the old bubble is replaced by the new one (only one visible).
    7. In the console confirm each action produces exactly one `[heed]` line per pipeline event with the { ts, sessionId, partnerId, event, data } shape, and that partnerId is "demo-platform" and sessionId is stable across the page load.
    8. Confirm the discount_offer postMessage: the harness's parent message listener (added in Plan 04-05) surfaces the heed:discount_offer payload when price_doubt fires — verify the payload shape and that no discount is actually applied by the SDK.
  </how-to-verify>
  <acceptance_criteria>

    - Single-bubble concurrency (D-05 replaced) works.
    - Each pipeline event logs exactly one correctly-shaped [heed] line; discount_offer postMessage payload is correct and grants nothing.
  </acceptance_criteria>

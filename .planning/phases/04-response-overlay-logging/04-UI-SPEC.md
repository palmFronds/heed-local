# Phase 4 — UI Design Contract

> Visual and interaction contract for Response Overlay & Logging. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Context note:** No `CONTEXT.md` exists for this phase (discuss-phase was not run) and this is not a typical app screen — it is a third-party embeddable overlay injected into an unknown host page (`repo2_heed_sdk.txt`, `response.js`). Every value below that isn't dictated by REQUIREMENTS.md/CLAUDE.md is a **reasoned default**, marked `(default)`, chosen for internal consistency with this repo's existing conventions (`test-harness/index.html`'s neutral palette, `src/inference.js`'s "hand-written, not abstracted" ethos). The planner/executor may override any `(default)` value; upstream-locked values (marked `(locked)`) must not be changed without a CONTEXT.md decision.

Applied inline (not via a `<style>` tag) — see Styling Mechanism below. |

### Styling mechanism `(default — flag to planner as an architecture decision)`

REQUIREMENTS.md's Out-of-Scope table explicitly rules out Shadow DOM for overlay isolation ("plain-div + pointer-events split is simpler and sufficient for v1"). Without Shadow DOM, a class-based `<style>` tag is vulnerable to the host page's own CSS cascade (a host rule like `div { color: red !important }` or a global `box-sizing` reset can silently corrupt the overlay's appearance). To stay isolated without Shadow DOM:

- Every overlay/response DOM node's presentation is set via **inline styles** (`element.style.cssText = '...'`), never via injected `<style>` + class selectors.
- Every declared property includes an explicit value (no reliance on browser/host defaults) — e.g. `box-sizing: border-box`, `line-height`, `font-family` are always set inline, never left to inherit from `document.body`.
- `element.style.setProperty(prop, value, 'important')` is used only where a specific collision risk is anticipated (documented per-property in the response element spec below), not blanket `!important` on every rule.

| Token | Value | Usage |
|-------|-------|-------|

| sm | 8px | Bubble internal padding (compact edge), gap between stacked copy lines |

| 3xl | 64px | Not used this phase |

- Dismiss control (`×`) hit target: **44×44px** minimum (iOS HIG touch target), even though the visible glyph is ~12px — achieved via padding, not a 44px visible box. `(locked)` — mobile-only signal set per CLAUDE.md means every tappable overlay element must clear iOS's minimum touch target or RESP-01's "individually tappable" success criterion is unverifiable in practice.
- `discount_offer`'s CTA button: 44px minimum height, same reasoning.

## Typography

Response bubbles are short (1–2 lines), so this contract uses compact sizing rather than page-body defaults.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Response body (tooltip/nudge_copy/social_proof/discount_offer main text) | 14px | 400 (regular) | 1.4 |
| Response emphasis (discount_offer CTA label, e.g. "See offer") | 14px | 600 (semibold) | 1.2 |
| Micro/meta (dismiss glyph, any fine-print under discount_offer) | 12px | 400 (regular) | 1.3 |
| Display | not applicable — the overlay never renders page-level headings; it renders exactly one short message per response instance |

`(default)` 3 declared sizes (14/14/12), 2 declared weights (400/600) — within the "3-4 sizes, 2 weights max" contract. Line-height 1.4 (not the generic 1.5 recommendation) chosen because bubbles are short, single-purpose strings, not paragraph text.

## Color

The overlay itself is the transparent, `pointer-events: none` full-viewport container (RESP-01) — it has no fill. The 60/30/10 split below applies to the **response bubble**, the only visible surface this phase renders.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#1a1a1a` | Bubble background fill (all 4 response types) |
| Secondary (30%) | `#ffffff` | Bubble text color; also the bubble's inner divider line at 12% opacity (`rgba(255,255,255,0.12)`) between main copy and discount_offer's CTA row |
| Accent (10%) | `#2f6fed` | Reserved for exactly two elements: (1) `discount_offer`'s CTA button background, (2) `tooltip`'s directional caret/pointer indicator. Nowhere else. |
| Destructive | not applicable this phase | No destructive actions exist in Phase 4 — dismiss is a neutral action, not a destructive one. Reserved token carried forward unused. |

`(default)` `#1a1a1a` reused verbatim from `test-harness/index.html`'s existing `button.primary { background: #1a1a1a }` / `body { color: #1a1a1a }` for cross-repo visual consistency — the overlay reads as "part of this system," not an unrelated third-party skin. `#2f6fed` is a new accent (this repo has no existing blue) chosen for sufficient contrast against both the dark bubble (`#1a1a1a`) and any host page background it may sit near.

Accent reserved for: `discount_offer` CTA button fill; `tooltip` caret/pointer glyph. Never used for dismiss controls, body text, or bubble borders.

Contrast check: `#ffffff` on `#1a1a1a` = 17.9:1 (passes AA/AAA for 14px text). `#ffffff` on `#2f6fed` (CTA label) = 4.6:1 (passes AA for 14px+ semibold text).

## Response Types — Copy & Intent Mapping

RESP-03 requires all 4 response types implemented, each "correctly for its mapped intent class." `repo2_heed_sdk.txt` and `admin/weights.js`'s cold-start mapping fix the 4 intent classes; the intent→response-type pairing below is `(default)` — not stated verbatim in any spec file — chosen so each response type addresses the psychological blocker its paired intent class represents.

| Intent class (from `inference.js`) | Response type | Copy (exact string) | Notes |
|---|---|---|---|
| `confusion` | `tooltip` | "Not sure what this means? Tap for a quick explanation." | Anchored to the signal's `bbox`; includes the `#2f6fed` caret pointing at the anchor element |
| `price_doubt` | `discount_offer` | Body: "Complete now and save on fees." CTA button label: "See offer" | Fires `postMessage` per shape below; Heed never grants/fulfills the discount itself (RESP-03) |
| `trust_gap` | `social_proof` | "Thousands of people completed this safely today." | No live counter, no fabricated precision — a static reassurance string, not a real-time stat (no backend exists to source a real number; a fake precise number would be a dark pattern) |
| `flow_friction` | `nudge_copy` | "Almost there — one more step to finish." | Brief reassurance; no CTA button, dismiss-only |

Every bubble also carries a dismiss control: glyph `×`, `aria-label="Dismiss"`, 44×44px hit target (see Spacing Exceptions).

### Positioning & clamping (RESP-02)

- `clampToViewport(bbox, bubbleSize)` `(default, prescriptive)`:
  - Horizontal: bubble `left = clamp(16px, bbox.left, viewportWidth - bubbleWidth - 16px)`. Bubble `max-width = calc(100vw - 32px)` (390px viewport → 358px max).
  - Vertical, safe-area-aware: usable top bound = `16px + env(safe-area-inset-top, 0px)`; usable bottom bound = `viewportHeight - 16px - env(safe-area-inset-bottom, 0px)`.
  - Default placement: bubble renders **below** the anchor (`bbox.bottom + 8px`). If `bbox.bottom + 8px + bubbleHeight` exceeds the usable bottom bound, flip to render **above** the anchor (`bbox.top - 8px - bubbleHeight`) instead.
  - If neither placement fits (anchor spans nearly the full viewport height), clamp the bubble to the usable bottom bound as a last resort — never allow it to render partially off-screen.
  - `env(safe-area-inset-*)` always has an explicit `0px` fallback — non-iOS/no-notch devices must not read `NaN`/`undefined` and crash positioning math.

### Animation `(default)`

- Entrance: opacity 0→1 + `translateY(8px)→0`, 200ms ease-out.
- Exit (manual dismiss or auto-timeout): opacity 1→0, 150ms ease-in, no transform.
- `prefers-reduced-motion: reduce` → opacity-only fade for both, no transform, same durations.
- Auto-dismiss timers: `tooltip` / `nudge_copy` / `social_proof` auto-dismiss after **6000ms** if not manually dismissed first. `discount_offer` does **not** auto-dismiss — it persists until the user taps the CTA or the `×` dismiss control, since it represents a decision point rather than a passive notice. Every auto-dismiss and manual dismiss alike fires exactly one `response_dismissed` log entry (see Logging Contract).

### `discount_offer` postMessage shape (RESP-03)

```json
{
  "type": "heed:discount_offer",
  "sessionId": "<string>",
  "partnerId": "<string>",
  "intent": "price_doubt",
  "timestamp": 1737100000000
}
```

- Sent via `window.parent.postMessage(payload, targetOrigin)` with an **explicit** `targetOrigin` read from config (never `'*'`) — RESP-03's "explicit target origin" requirement. Exact config field name (e.g. `config.partnerOrigin`) is an implementation detail for the planner; this contract fixes only the payload shape and the "never wildcard origin" rule.
- Heed's own code contains no logic that grants, applies, or fulfills a discount — the payload only signals that the offer moment exists; the host page owns fulfillment entirely.

## Copywriting Contract

Template's default roles (Primary CTA / Empty state / Error state / Destructive confirmation) are adapted below — this phase has no page, no empty-data state, and no destructive action; the equivalent surfaces are the response bubble's CTA and its two failure-adjacent states.

| Element | Copy |
|---------|------|
| Primary CTA (discount_offer only — the sole response type with a button) | "See offer" |
| Dismiss control (all 4 response types) | `×` glyph, `aria-label="Dismiss"` — no text label, universal across types |
| "Empty state" equivalent — no response fires | N/A by design: below the 0.65 confidence threshold (INF-03) or outside `activeScreens`, the overlay renders nothing and logs nothing. There is no visible empty state to design; silence is the correct behavior. |
| "Error state" equivalent — malformed/missing bbox on the triggering signal | Response still renders, anchored via the last-resort viewport-bottom clamp (see Positioning above) rather than failing silently or throwing — a rendering fallback, not a user-facing error message. No error copy is ever shown to the end user; a partner-facing SDK never surfaces internal failures as end-user-visible text. |
| Destructive confirmation | Not applicable — no destructive actions exist in this phase (dismiss is non-destructive and requires no confirmation) |

## Logging Contract (LOG-01)

Every one of the 6 pipeline events produces exactly one line: `console.log('[heed]', JSON.stringify(entry))`, `entry = { ts, sessionId, partnerId, event, data }`. `(locked)` — verbatim from REQUIREMENTS.md LOG-01 and `repo2_heed_sdk.txt`.

| `event` | Fires when | `data` payload (interaction-relevant fields) |
|---|---|---|
| `signal_detected` | A signal payload arrives on the bus (already implemented upstream in Phase 2/3 — this phase only adds the remaining 5 event types around it) | `{ type, targetSelector, bbox, timestamp }` (existing shape, unchanged) |
| `inference_run` | `inference:result` is published, regardless of `fires` | `{ intent, confidence, fires }` at minimum |
| `response_fired` | A response bubble is actually rendered (i.e. `fires === true` AND `activeScreens` gate passes) | `{ intent, responseType, targetSelector }` |
| `response_dismissed` | The bubble is removed — either by manual `×` tap, CTA tap, the 6000ms auto-timeout, or being replaced by a new above-threshold response | `{ responseType, dismissReason: "manual" \| "cta" \| "timeout" \| "replaced" }` |
| `flow_complete` | `completionSelector` becomes visible (mirrors existing `flowComplete` detection in `signal.js`) | `{}` or omitted extra fields — event name alone carries the meaning |
| `flow_abandoned` | Session ends (e.g. `back_intent` fires with no subsequent completion, or tab/session teardown) without `flow_complete` having fired | `{}` |

- All 6 event types are emitted **only from the logging layer** (`log.js` per `repo2_heed_sdk.txt`'s file structure) — no other module calls `console.log('[heed]', ...)` directly. `(locked)`.
- No log line fires while the current pathname is outside `config.activeScreens` (mirrors Phase 6's INTEG-01 acceptance check "no logs fire on Screen 1").
- `dismissReason`'s 4th value, `"replaced"`, was added per `04-CONTEXT.md` D-05 (single-bubble concurrency): a new above-threshold `inference:result` arriving while a bubble is showing dismisses the existing bubble with this reason before rendering the new one.

<sc2_verification_decision>
## SC2 verification (RESEARCH Open Question #2 — resolved)
SC2 ("restarting the harness after a persisted file exists loads learned weights, not cold-start") is verified by **the soak-test script's before/after GET-readback comparison in Plan 05-05**, NOT by a live-receiver Playwright e2e case. Rationale: playwright.config.js has no `webServer` block and navigates `file://` (opaque origin), so a real-browser SC2 test would require standing the receiver up alongside Playwright and adding a debug hook for the active weights vector — flaky and heavyweight. The soak-test reads the persisted weights back through the receiver's exact GET /weights endpoint (the same endpoint this harness bootstrap fetches) and runs forwardPass against them, deterministically proving a cold-start restart would load learned (non-cold-start) weights. This plan (05-04) supplies the runtime mechanism; Plan 05-05 supplies the gate. The harness bootstrap's own fetch-failure fallback (SC4 harness-side) is proven at the unit level by tests/index.test.js's `cold-start fallback` case (authored 05-01, made GREEN in 05-03), since a failed harness fetch reduces to the bare `initDemo()` cold-start path.
</sc2_verification_decision>

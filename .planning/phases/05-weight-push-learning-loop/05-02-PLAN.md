  <action>
    Create local-receiver/server.js as an ES module using ONLY Node built-ins (node:http, node:fs, node:path, node:url) — no npm import, no Express (D-04). Start from RESEARCH Architecture Pattern 3 verbatim, adapted for testability: export a factory `createReceiver({ weightsPath = DEFAULT_WEIGHTS_PATH } = {})` that returns the http.Server WITHOUT calling .listen(); resolve DEFAULT_WEIGHTS_PATH to local-receiver/weights.json via fileURLToPath(import.meta.url). Export `isValidWeights(w)` as a boolean-returning validator mirroring validateWeightsShape's 4x4-matrix + 4-vector numeric-finite checks (D-06 — returns false, never throws). Handlers: OPTIONS → 204 CORS preflight short-circuit; GET /weights → read the file, on ENOENT 404, on parse error or isValidWeights===false respond non-2xx WITHOUT crashing (never serve garbage), on valid serve the raw validated bytes; POST /weights → accumulate the body with a MAX_BODY_BYTES cap (64*1024) calling req.destroy()+413 on overflow, then JSON.parse the raw buffer REGARDLESS of Content-Type (D-03/Pitfall 3), reject with 400 if parse throws or isValidWeights is false (never write on invalid), else write to a temp file then fs.rename onto weightsPath (atomic swap) and 200. Set `Access-Control-Allow-Origin: *` with NO Access-Control-Allow-Credentials (file:// opaque origin, Pitfall 5). Add a req.on('error') handler so an aborted/malformed request stream never throws into the process. Guard the actual `.listen(process.env.PORT || 4310)` behind a run-as-main check (compare import.meta.url to the executed argv path) so importing the module in tests does not bind a port. Header comment: dev/test-only tooling, run via `npm run receiver`, never imported by src/ or bundled into dist/sdk.js, never a production endpoint (CLAUDE.md scope).
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (file:// harness) → receiver | Untrusted, opaque-origin POST/GET crosses into the local Node process. |
| receiver → OS filesystem | Weight bytes are written to a fixed local path (never derived from request input — no path-traversal surface). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Denial of Service | POST /weights body parsing | medium | mitigate | MAX_BODY_BYTES (64KB) cap with req.destroy()+413 on overflow, try/catch around every JSON.parse, and a req.on('error') handler — a malformed/oversized/aborted body yields a 4xx and never throws into the process (RESEARCH Pattern 3). |
| T-05-02 | Tampering | GET /weights serving on-disk file | medium | mitigate | isValidWeights() re-validates the on-disk shape on every GET; a corrupt/tampered file is never served as-is (non-2xx / last-known-good), and SDK-side validateWeightsShape() is a second independent check (D-06). |
| T-05-03 | Spoofing | Access-Control-Allow-Origin: * | low | accept | Wildcard CORS lets any page open in the same browser POST to localhost:4310 while the receiver runs — accepted as local-dev-only tooling per CLAUDE.md "no production endpoint" scope; MUST NOT be carried into any future production weight-push design. |
| T-05-04 | Information Disclosure | persisted weight bytes | low | mitigate | The body is numeric {W1,b1,W2,b2} only — no PII, no field values, no identity (CLAUDE.md No-PII); isValidWeights rejects any non-numeric shape before write. |
| T-05-SC | Tampering | npm/pip/cargo installs | low | accept | Zero new packages (Node built-ins only, D-04) — no install step, no supply-chain surface. |
</threat_model>
